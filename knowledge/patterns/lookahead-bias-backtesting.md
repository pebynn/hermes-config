# P004: 量化回测中的前瞻偏差（Look-Ahead Bias）

## 特征

1. 回测结果异常优秀（年化 >50%、胜率 >70%），但实盘/净化后骤降为负
2. 信号来源依赖公告前/事件前的量价数据
3. 去除信号中的未来信息后，策略完全失效
4. 多次迭代无改善（R2→R5 连续退化）

## 实例

### 实例 1: 策略B PEAD 公告前量比信号 (2026-05)
- **阶段**: R2-R4 虚高 → R5 净化后-12.67%年化/0%胜率/12笔全败
- **根因**: 公告前量比数据含未来函数（公告公布时量比已包含公告发布后的交易量）
- **事故**: 4轮迭代投入，最终宣告不可达
- **处理**: 终止独立进化，降级为 alpha 分量

### 实例 2: V4 资金流生命周期策略 (2026-04)
- **阶段**: 设计阶段即发现核心数据存在前瞻偏差
- **根因**: 北向资金流数据2024-08停更，主力资金流API含日内累计（不能用于日线回测）
- **处理**: 标记为「已确认死路」，设计阶段终止

## 根因

量化回测中最危险的偏差类型之一：
1. **数据含未来信息**：公告前的量比数据实际包含公告发布后的信息
2. **API 数据的时间对齐问题**：日内累计数据不能直接用于日线策略
3. **停更数据的历史快照**：使用最新快照回测历史时期，信息泄露

## 修复/预防

1. **回测前必须做前瞻偏差检测**：
   - 检查信号计算是否使用了目标日期之后的信息
   - 逐日模拟（walk-forward）而非全量回测
2. **策略设计阶段即检查数据源**：
   - API 返回的数据字段的时间语义是否清晰
   - 是否存在"已总结的统计量"（如日内累计）
3. **独立验证门禁**：
   - 所有新策略在集成前必须经过净化回测（removing any look-ahead data）
   - 净化前 vs 净化后的胜率/夏普差异 >30% → 🚨前瞻偏差告警
4. **与 D002 B+D 层联动**：
   - pre_kanban_create.py 注入前瞻偏差检查提醒
   - 策略类 kanban 任务自动附加"确认无未来函数"检查项

## 跨域适用

- **finance-domain**（主要域）：所有量化策略
- **writing-domain**：数据采集脚本的时间对齐检查
- **code-domain**：任何涉及时间序列数据的模块
