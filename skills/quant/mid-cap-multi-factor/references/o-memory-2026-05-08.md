# OOM 与 N_WORKERS 调优 — 2026-05-08

## 环境限制

- 系统总内存: 31GB
- 空闲可用: ~2.5GB (其余被各种服务占用)
- quant_env Python: pandas/numpy/pyarrow 较重

## signal_engine 内存行为

实测 generate_signal 8 worker:
- 每 worker 加载自己的一份股票列表/行业映射 (~30MB)
- 加载 K线 parquet 到 DataFrame (每只~0.5-2MB)
- 8 worker + 父进程 = 9个Python进程, 每进程~400MB-1GB
- 峰值内存 ~6-8GB → 触发 OOM killer 或 swap 风暴 (磁盘IO飙升, 扫描变慢10x)

## 推荐配置

| 场景 | N_WORKERS | 预期耗时 | 内存 |
|:----|:---------|:---------|:----|
| 手动调试 | 2 | ~12min (618只) | <3GB |
| 手动扫描 | 4 | ~6min (618只) | <5GB |
| cron 环境 | 2-3 | ~8-10min | <4GB |
| 带force的全量 | 1 | ~25min | <2GB |

## 修改点

- `daily_signal_report.py`: `N_WORKERS = 8` → `4`
- `mid_cap_strategy.py`: `generate_signal`内部默认值
- `signal_engine.py`: 无全局worker数, 在调用处控制

## 诊断

```bash
# 检查是否有残留worker
ps aux | grep signal_engine | grep -v grep | wc -l

# 检查内存
free -h | grep Mem

# 如果>8个worker, 批量杀
pkill -9 -f signal_engine
```
