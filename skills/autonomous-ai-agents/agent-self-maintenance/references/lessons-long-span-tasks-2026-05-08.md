# 会话教训: 2026-05-08 — 长跨度任务执行 & 用户交互模式

## 🔴 CRITICAL: 用户说"不需要问" = 直接动手，停止请示

用户明确说了"我不需要你来问我，这个是你需要解决的问题"（2026-05-08 长跨度任务会话）。

模式识别：当用户给出清晰方向（"深度调研后给我一个合适的方案"、"这是你需要解决的问题"），这意味着：
1. 他不需要中间决策参与
2. 他只需要最终结果
3. 停止追问和确认

正确反应：直接设计方案并执行。先出完整方案（含代码/脚本/配置变更），执行完汇报。不要问"要不要这样"、"这样做可以吗"、"要试吗"。

## 🟠 HIGH: 5阶段计划不靠谱 — 用 pipeline 替代决心

用户质疑"你列了5个阶段，而且时间跨度那么长，你真的能完美执行下来吗？如果不能完美执行完，那不就只动了一个方案A吗"（2026-05-08）。

根本问题：我倾向于提出多阶段计划（方案A→B→C），但执行依赖"我记住"和"我不被中断"。这两个前提都不成立。

正确模式：1天内能闭环的任务 → 直接做。跨天的 → 设计 pipeline（每阶段独立脚本 + cron 自动推进）。不要提出依赖于我个人执行力的计划。

## 🟠 HIGH: 用户给的架构思路必须立即实现

用户说"我昨天给提的思路你好像忘了或者根本就没当回事"（2026-05-08 关于daily继承任务）。

用户提出工程设计思路（如"daily自动继承未完成任务+计时"）时，这是操作指令，不是闲聊。必须立即：
1. 评估可行性
2. 落地为脚本/SOUL.md规则/MCP工具
3. 通知用户已落地
不要只存 memory。

## 🟡 MEDIUM: 长跨度任务的正确解法

用户纠正了3次（"长跨度任务怎么做"、"你没有把...整明白"、"我不需要你来问我"）后最终明确：

正确解法不是"我坚持执行"，而是 "cron pipeline"：
1. 长任务拆为 self-contained 阶段
2. 每个阶段写独立脚本（幂等、可验证）
3. 提交到 pipeline_runner.py
4. cron 自动推进，不受会话切换影响
5. L3 决策点 pause，等用户
6. 用户 resume → 继续

这个模式的核心转变：我从"执行者"变为"执行机制的设计者"。
