# 跨域去重合并方法论

从 finance-domain × writing-domain 双域审计+合并实战中提炼的通用模式。

---

## 适用场景

- 两个或以上的域共享数据源/脚本/cron/技能
- 系统膨胀感——脚本数/代码行/维护点过多
- 用户要求"降低系统臃肿度"

---

## 三阶段流程

```
Phase 1: 双域并行审计 → Phase 2: 交叉重叠分析 → Phase 3: 结构化合并方案
```

### Phase 1: 双域并行审计

双域同时 delegate（dispatch-parallel-agents），各自产出：

| 产出项 | 具体内容 |
|:-----|:---------|
| 配置文件 | SOUL.md / config.yaml 状态 |
| 脚本清单 | 全量 .py 列表，含路径+行数+功能描述 |
| Cron管线 | 全量 cron jobs，含状态/最后运行/错误 |
| 数据管线 | API源矩阵、数据流架构、已知风险 |
| 数据库/存储 | 表结构/缓存/文件格式/规模 |
| Skills引用 | 域声明的技能清单+实际使用情况 |

产出一个结构化MD报告，格式统一方便后续对比。

### Phase 2: 交叉重叠分析

从5个维度逐一比对两份审计报告：

#### 2.1 数据源重叠
- 列出各域使用的所有数据源（API/数据库/文件）
- 标记共享源（如两个域都用AKShare）
- 计算"引用密度"（总引用次数 vs 共享次数）
- 判断合并可行性：高密度共享 → 可提取统一适配层

#### 2.2 存储层重叠
- 对比存储格式（MySQL vs JSON vs parquet）
- 检查共享凭证文件（cookies/tokens）
- 判断是否适合统一存储引擎

#### 2.3 脚本层重叠（核心）
- **逐字节比对**：同名文件计算MD5，标记100%一致的拷贝冗余
- **逻辑比对**：功能相近但结构不同的脚本（如两个audit_guard版本）
- **隐式重叠**：同一底层能力的不同封装（如各自独立调用AKShare）
- **跨域引用**：A域脚本是否import/call了B域脚本

#### 2.4 Cron管线重叠
- 合并两份cron清单，检查ID重复
- 如果两域共用同一批cron ID → 它们事实共享管线
- 合并收益在脚本层面而非cron层面

#### 2.5 Skills重叠
- 列出两域SOUL.md中声明的技能
- 标记可跨域复用的技能（如deep-research/web-researcher）
- 标记已跨域但未规范化的引用

### Phase 3: 结构化合并方案

使用三级优先级分类，每项合并用固定格式：

```
P0 — 立即合并（零风险，纯去重）
P1 — 本周合并（需要适度重构）
P2 — 需要设计（跨月推进）
```

#### 合并项模板

| 项目 | 内容 |
|:-----|:-----|
| 当前状态 | 描述现状，量化指标（文件数/行数/差异类型） |
| 目标状态 | 清晰的一句话目标 |
| 合并方式 | 分步操作（1/2/3步），每步可执行 |
| 影响范围 | 受影响的脚本/文件/cron列表 |
| 收益 | 量化收益（减少行数/脚本数/维护点） |
| 风险 | 风险等级+降级措施 |

#### 收益预估汇总表

| 优先级 | 项数 | 脚本减少 | 代码行减少 | 维护减轻 |
|:------:|:----:|:--------:|:---------:|:--------:|
| P0 | N | -X | -Y | 去重 |
| P1 | N | +X/-Y | -Z | 统一接口 |
| P2 | N | 0/-X | +Z/-W | 长期优化 |

## 关键发现模式

在finance×writing实战中发现的重要规律：

### 发现1: Cron管线已事实共享
两个域审计列出了**完全相同的cron job ID**——它们不拥有独立管线，而是在同一条15:30→16:00→21:00流水线上各自消费。合并收益在脚本去重，不在减少cron数。

### 发现2: 数据桥比存储统一更务实
finance用MySQL+parquet（量化回测），writing用JSON文件（写作素材）。强制统一存储引擎收益低风险高。更好的方案是建立**单向数据桥**：finance产出 → writing消费（通过标准化接口），各自保留最优存储格式。

### 发现3: 跨域引用需规范化
当前跨域调用方式混乱：subprocess硬编码路径、SKILL.md建议但未实现、import路径假设。统一后在共享库中定义清晰契约。

## 实施路线图模板

```
第1天 P0        第2-5天 P1          第2-4周 P2
├─ 删除冗余副本  ├─ 提取共享模块     ├─ 统一仪表板
├─ 统一双版本    ├─ 建立数据桥       ├─ 交叉验证
└─ 标准化路径    └─ 修复边缘问题     └─ 长期架构调整
```

---

## 注意事项

1. **先验证MD5再删除**：`md5sum file1 file2` 确认完全一致后再动手
2. **删除前确认引用**：`grep -r "import xueqiu_kline"` 确认所有调用方路径
3. **统一后立即跑一次管线**：验证去重没有破坏数据流
4. **Cookie/凭证统一管理**：共享的雪球cookie/api key只在一处维护+过期预警
5. **保留回滚能力**：删文件前做备份或确认git有历史版本
