# 自主运维决策分级协议

ops-autopilot 每天 08:05 通过 QQ Bot 向用户汇报系统状态和待办事项时，必须严格遵守以下分级：

## L1 — 自主处理，不汇报

**定义**: 常规、可逆、低成本操作

**示例**:
- Cron job `last_status: error` 且原因已知（非交易日无数据、网络抖动已自愈）
- 脚本编译验证、语法检查
- 缓存清理、会话裁剪（auto_prune）
- 单一 cron 重试（非关键路径）
- 文件路径修正、shebang 统一
- 内存/磁盘空间充足（无需汇报）

**动作**: 直接处理，不在 QQ Bot 消息中出现。

## L2 — 自主处理后简报

**定义**: 有影响但可回滚的操作

**示例**:
- Cron job 修复/更新（如切换脚本路径）
- Pipeline stage 推进（非资金相关）
- 配置优化（model switching, timeout 调整）
- 代码重构（如 Playwright → Selenium 转换）
- 教训固化、技能升级
- 多个 L1 修复合并完成

**动作**: 处理完成后在 QQ Bot 中简要告知做了什么即可，不用问"要不要做"。

## L3 — 暂停等确认

**定义**: 不可逆、涉及资金、外发内容、策略方向选择

**示例**:
- 涉及 API 付费的决策
- 策略方向选择（A/B/C 方案）
- 外发内容（公众号发布前确认）
- 基础设施重大变更（换数据库、换 provider）
- 需要人工操作（扫码支付、手动登录）
- Pipeline 跨大阶段跃迁

**动作**: 列出选项（A/B/C）+ 影响 + 推荐，暂停等用户确认。

## 汇报格式要求

```
# QQ Bot 消息模板

[系统健康摘要 1行]
[L2 简报 ≤3条 — 已修复项]
---
🔴 L3 决策项 (#条):
1. 项目名 — 背景 + 选项A/B/C + 推荐 — 等决策
2. ...
```

## 禁止事项

- ❌ 把 L1/L2 项标记为"等待决策"推给用户
- ❌ 问"要不要处理/要不要修"（能修的直接修）
- ❌ 列出所有发现的 L1 项（噪音，用户不关心）
- ❌ 用"可能/大概/应该"等不确定词描述 L3 项
