# T1-C-R2: 修复缠论v2 — 实施计划

## 问题诊断

### 1. Lookahead bug (结构性错误) — 🔴 P0
**位置**: strategy_chan_v2.py 第402行
```python
end_dt = rd_dt + timedelta(days=2)  # BUG: 使用T+2日数据选股
```
- 回测在调仓日rd_dt选择入场，但允许检查到rd_dt+2的buy2信号
- 这导致"预知未来"：知道了接下来2天的信号才入场
- R23 (strategy_chan.py) 同样存在此bug（第403行）
- **修复**: `end_dt = rd_dt`，入场只能基于调仓日当天及之前已知数据

### 2. 北交所代码未过滤 — 🟠 P1
- T3审查发现11/88笔交易(12.5%)涉及920xxx北交所代码
- 当前stock_list.parquet无920xxx代码（只有00/60/30/68）
- 但不排除未来或其他数据源引入
- **修复**: 在universe构建时显式过滤4xxxxx/8xxxxx/920xxx前缀

### 3. Sina数据源不足 — 🟡 P2
- Sina缓存仅100只股票，0笔交易
- **修复**: 启动时若source=sina且股票数<500，打印警告并建议切换mysql

### 4. v2 vs R23 性能崩塌分析
R23 (558%/47.9%wr) → v2 (71%): 差异源于两方面:

| 参数 | R23 | v2 | 影响 |
|:------|:----|:---|:---- |
| START_DATE | 2026-01-01 | **2025-05-01** | R23只跑牛市4个月，v2覆盖2025熊市全年 |
| MAX_HOLD_DAYS | 30 | 15 | 持仓时间减半，难以穿越回调 |
| STOP_LOSS | -8% | -6% | 更紧止损→更多止损出局 |
| TAKE_PROFIT | 10% | 8% | 更低止盈上限 |
| MAX_HOLDINGS | 8 | 5 | 更集中→单笔风险更大 |
| REBALANCE | 7天 | 5天 | 更高频交易 |

**根因**: v2扩展了日期范围（包含2025熊市）+ 同时收紧所有参数 = 双重打击

## 设计方案

### 方案选择: 渐进式修复（推荐）
1. 修复lookahead（必须）
2. 添加北交所过滤（必须）
3. 添加Sina数据源警告（必须）
4. 参数回退：部分回归R23参数以恢复性能
   - STOP_LOSS: -6% → -8%（减少止损出局）
   - MAX_HOLD_DAYS: 15 → 25（给持仓更多时间）
   - 保留v2的集中持仓特性（4-5只）

### 替代方案（放弃v2，直接回退R23）
- 优点: 直接恢复558%性能
- 缺点: 丢弃v2迭代工作
- **不采用**: 任务要求优化v2而非回退

### 风险
- 修复lookahead会降低回测收益（不再"作弊"）
- 参数调整需在修复lookahead后重新评估
- 2025年熊市期间无论参数如何都难以盈利
