## 设计方案

### P0: 修复概念数据获取 — 添加akshare第三fallback

**问题根因分析**:
- `_load_concepts_via_eastmoney()` (strategy.py:330-396): 使用urllib直接调用EastMoney API
  - IPv4强制已实现（socket.getaddrinfo monkey-patch），但有IP限流/封禁风险
  - EastMoney API可能因反爬策略返回空数据
  - R2审计确认: mcp_api_working=false, eastmoney_fallback=false
- `_load_concepts_via_mcp()` (strategy.py:230-328): MCP subprocess通信
  - 依赖 `~/.hermes/node/lib/node_modules/stock-sdk-mcp/dist/index.js` 二进制文件存在
  - subprocess通信链路长，容易超时/崩溃

**方案**: 三级fallback链路 MCP → EastMoney → akshare

**新增方法**:
1. `_load_concepts_via_akshare() -> list`:
   - 调用 `ak.stock_board_concept_name_em()` 获取全量概念列表
   - DataFrame包含列: `代码`(BKxxxx), `名称`, `涨跌幅` 等
   - 按涨跌幅绝对值排序，取top N个概念的代码
   - 异常时返回空列表

2. `_load_constituents_via_akshare(concept_code: str) -> list`:
   - 调用 `ak.stock_board_concept_cons_em(symbol=concept_code)` 获取成分股
   - DataFrame包含列: `代码`(股票代码), `名称` 等
   - 提取股票代码列表返回
   - 异常时返回空列表

**修改现有方法**:
- `_load_top_concepts()` (line 219-228): 在EastMoney fallback后添加akshare
  ```
  MCP → EastMoney → akshare → []
  ```
- `_load_concept_constituents()` (line 398-406): 同上三级链路
  ```
  MCP → EastMoney → akshare → []
  ```

**缓存策略**: akshare调用也走 `~/.finquant/cache/concepts/` 缓存目录，TTL=24h

**替代方案考虑**:
- ~~THS iFinD概念数据~~: 需要付费API，不适合
- ~~新浪概念板块~~: 接口不稳定，已废弃
- ✓ akshare: 开源免费，API稳定，已在环境中安装(v1.18.60)

**风险**:
- akshare首次调用可能较慢（~5-10s），需设置合理超时
- 缓存命中后延迟可忽略
- akshare API格式变更风险（低，使用稳定版本）

### P1: 月度调仓2天分批建仓窗口

**问题**: 月度调仓日仅当天可入场（≤3只/天），6个月仅6批次，理论最大18笔。R2实际仅15笔，过于保守。

**方案**: 调仓日+次日作为入场窗口

**实现细节** (backtest_engine.py):
1. 在 `DEFAULT_CONFIG` 新增参数: `"rebalance_entry_window": 2`（调仓窗口天数）
2. 在 `__init__` 中初始化: `self._last_rebalance_date = None`
3. 在 `run()` 主循环的调仓块（line 272）记录 `_last_rebalance_date`
4. 在调仓块之后，新增"窗口入场块"：
   - 条件: `_last_rebalance_date` 存在 且 当前日期在窗口内 且 非调仓日本身
   - 行为: 仅入场（不旋转持仓、不重新选股），最多3只新股
   - 窗口判断: `cur_date` 在 sorted `backtest_dates` 中的索引 减去 `_last_rebalance_date` 索引 < `rebalance_entry_window`

**关键**: 
- 窗口日不触发 `_select_stocks()` — 避免覆盖月度选股结果
- 窗口日不旋转持仓 — 仅填补空缺
- 窗口日仍受 `max_new_per_batch` 限制（≤3只/天）

**替代方案考虑**:
- ~~修改 is_rebalance_day 返回True两天~~: 会导致第二天也触发完整的调仓逻辑（含旋转），不精确
- ✓ 独立窗口入场块: 精确控制行为，不影响调仓语义

**配置新参数**:
```python
"rebalance_entry_window": 2,  # strategy.py STRATEGY_C_CONFIG 中设置
```

### P2: 大盘趋势过滤 — 上证指数>MA20才入场

**问题**: 40%胜率距45%目标差5pp。熊市中策略无方向性优势，增加趋势过滤可减少无效交易。

**方案**: 入场前检查上证指数是否在MA20之上

**实现方式**:
1. 在 BacktestEngine 新增可选钩子方法 `_should_enter_market(cur_date, xsec_today)`:
   - 基类默认返回 True
   - StrategyC 覆写以添加趋势过滤

2. StrategyC 实现:
   - 新增 `_index_data` 属性 (pd.DataFrame)
   - 新增 `_load_index_ma20(cur_date)` 方法:
     - 首次调用时通过 akshare 加载上证指数日线数据
     - 计算20日均线
     - 缓存结果
   - `_should_enter_market()`: 比较当日指数收盘 vs MA20

3. 在 backtest_engine.py 入场块（line 295-314）调用 `_should_enter_market()`

**索引数据获取** (`_load_index_ma20()`):
```python
import akshare as ak
df = ak.stock_zh_index_daily(symbol="sh000001")
# 筛选日期范围
# 计算 MA20
# 缓存到 self._index_ma20_cache
```

**替代方案考虑**:
- ~~使用 is_bear_market(MA20<MA60)~~: 过于严格，MA20过滤更适合A股短期趋势
- ~~MCP获取指数数据~~: MCP不可靠，与P0问题相同
- ✓ akshare获取指数: 可靠，本地已有
- ~~sina/mysql数据源~~: 索引数据可能不在股票数据中

**风险**:
