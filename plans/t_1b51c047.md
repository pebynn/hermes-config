## 2. 设计方案

### 方案选择：回归 R9/R10 基线 + 行业映射修复 + MySQL 优先

**选择的方案**：以 R9 参数为基线（strategy_reversal.py），修复行业映射后逐步优化。

**替代方案**：
- 方案A: 直接修改 v2 参数调优 — 风险高，R9→R11 的改动方向错误
- 方案B: 完全重写 — 不必要，核心引擎 (ReversalBacktest) 设计合理

**风险**：
- MySQL 连接可能在运行时不可用 → 保留 Sina fallback
- R9 参数在更大股票池下可能表现不同 → 预留参数调优空间

### 核心改动

```
改动 1: load_industry_map() → 使用 data_common.get_industry_map()
  理由: data_common 已有完整实现，直接从 stock_list 缓存读取 4939 条行业映射
  影响: 行业覆盖从 0.4% → 100%

改动 2: 数据源优先级 MySQL > Sina
  理由: MySQL 全A股 5000+ 只 vs Sina 仅 100 只
  实现: --source mysql (default), --source sina (fallback)

改动 3: 参数回归 R9/R10 基线
  理由: R9 参数经多次迭代验证（年化 75%+ 全A股无杠杆上限）
  具体: 见下方参数对比表

改动 4: 延长持仓周期
  理由: 减少止损退出频率 (33%→目标<20%)
  具体: REBALANCE_DAYS 2→3, TIME_STOP_DAYS 4→5

改动 5: 清理参数重复定义
  理由: 当前文件有 2 组参数定义（L55-58 和 L79-93），后者覆盖前者
  修复: 合并为单一参数区
```
