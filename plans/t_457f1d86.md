## 1. Overview

Build a zero-token, no_agent cron-compatible script that monitors B/D layer
injection rates for degradation trends, inspired by the EvoClaw benchmark
(arXiv 2603.13428) which shows agent performance can degrade >80%вҶ’38% under
continuous evolution.

## 3. Data Source

### Primary: kanban.db (SQLite)

B-layer rate = tasks with `body LIKE '%е·ІзҹҘйҷ·йҳұ%'` / total tasks per day
D-layer rate = task_runs with `summary LIKE '%[LESSONS]%'` / completed tasks per day

Query mirrors `audit_bd_layer.py:scan_kanban_db()` but extends to 7-day window
with per-day grouping via `date(created_at, 'unixepoch')`.

### Secondary: History File

`~/.hermes/metrics/evoclaw_history.json` вҖ” persistent array of daily snapshots:
```json
[{"date":"2026-05-13","b_count":5,"b_rate":0.71,"d_count":2,"d_rate":0.29,
  "total":7,"done":7,"checks":["ok"],"alerts":[]}]
```

## 4. Degradation Detection Logic

### 4.1 3+ Consecutive Days Decline (вҶ’ P1 Warning)

For each metric (B_rate, D_rate):
- Compare today's value vs yesterday's
- If today < yesterday AND yesterday < day-before AND day-before < 3-days-ago
  вҶ’ "3+ consecutive days decline" triggered
- Send QQ Bot P1 warning with trend data

### 4.2 >50% Drop vs Baseline (вҶ’ P0 Alert)

Baseline = average of days -7 through -3 (5-day window, excludes yesterday and today)
- If B_rate today < baseline_b * 0.5 вҶ’ P0 alert
- If D_rate today < baseline_d * 0.5 вҶ’ P0 alert
- Send QQ Bot P0 alert immediately

### 4.3 Edge Cases

| Scenario | Behavior |
|----------|----------|
| < 3 days of history | Skip degradation check, save snapshot only |
| No new tasks today | Exit 0 with "no data" message |
| kanban.db missing | Exit 2 with error to stderr |
| Zero completed tasks | D_rate = 0.0, still compute B_rate |

## 10. Design Decisions

### Choice: SQLite direct query vs audit_bd_layer.py output file

**Decision**: Query kanban.db directly.

**Rationale**:
- audit_bd_layer.py does not persist its output (only prints to stdout)
- Adding a persistence layer to audit_bd_layer would couple two independent scripts
- Direct kanban.db query is idempotent, fast (~ms), and source-of-truth
- The computation mirrors audit_bd_layer's logic exactly

**Alternative considered**: Have audit_bd_layer.py write JSON output to a known
path, then evoclaw script reads it. Rejected because it adds write-side-effects
to audit_bd_layer and creates a dependency that could break independently.

### Choice: Python import notify vs shell out

**Decision**: Use `from notify import send` (Python import).

**Rationale**:
- Both scripts live in the same directory (~/.hermes/scripts/)
- Faster (no subprocess overhead)
- Better error handling (can catch ImportError explicitly)
- notify.py is designed as a module (`def send(...)`)

### Choice: Baseline definition for 50% drop

**Decision**: Baseline = average of days -7 through -3 (5-day window).

**Rationale**:
- Excludes today (the target under test)
- Excludes yesterday (may be the start of the decline)
- 5-day window provides stable reference without being too stale
- EvoClaw paper observed degradation over days, so comparing today vs
  recent-stable-baseline is the right pattern

**Alternative considered**: Fixed historical baseline. Rejected because
B/D rates naturally fluctuate вҖ” we want to catch sudden drops, not gradual
seasonal changes.

## 11. Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| kanban.db schema changes | Low | Use only stable columns (tasks.body, task_runs.summary) |
| notify.py import fails | Low | Add try/except + stderr fallback |
| History file corruption | Low | JSON parsing wrapped in try/except, auto-repair by rewrite |
| False P0 alerts on low-data days | Medium | Skip alert if today's total < 3 tasks |
| Cron fires before kanban.db is written | Low | Schedule at 3am, well after daily batch |
