# R11-C 代号优化实施计划

## 任务概要
优化 strategy_chan_C.py：提升胜率至 >45% 同时保持年化 >40%

## 上轮(R10-C)基线
- ann_return: 41.97%, win_rate: 38.5%, sharpe: 0.48, mdd: -16.26%
- 胜率从 R9 的 40.5% 降到 38.5%，量比过严(3.5)是主因

## R11-C 变更设计

### 变更 1: VOL_RATIO_MIN 3.5 → 3.0
- **文件位置**: strategy_chan_C.py:65
- **实现**: 修改常量值 `VOL_RATIO_MIN = 3.0`
- **理由**: R10 量比从 2.5 收紧到 3.5 导致胜率下降 2pp，回到 3.0 缓解过过滤
- **替代方案**: 3.25（折中），但任务指定 3.0
- **风险**: 可能引入低质量交易，但其余 4 条件(MA60斜率, MA20>MA60, 底分型, 动量)仍能过滤

### 变更 2: 新增 20 日动量确认 (>0)
- **文件位置**: strategy_chan_C.py (precompute_indicators + check_entry_conditions)
- **实现**: 
  1. `precompute_indicators()` 新增 `mom_20d` 指标
  2. `check_entry_conditions()` 新增条件：`mom_20d[i] > 0`
  3. 新配置: `MOM_20D_ENABLED = True`
- **伪代码**:
  ```python
  # In precompute_indicators:
  mom_20d = np.full(n, np.nan)
  for i in range(20, n):
      if close_v[i-20] > 0:
          mom_20d[i] = close_v[i] / close_v[i-20] - 1
  # In check_entry_conditions:
  if MOM_20D_ENABLED:
      if np.isnan(ind["mom_20d"][i]) or ind["mom_20d"][i] <= 0:
          return False
  ```
- **理由**: 确保短期趋势方向正确，补充 MA20>MA60 的相对位置判断
- **替代方案**: 5日动量（太敏感）/ 60日动量（太滞后），20日折中
- **风险**: 进一步减少候选数量，但配合量比回调 3.5→3.0 可抵消

### 变更 3: 行业热度过滤 (借鉴 Strategy B SECTOR_HEAT)
- **新增函数**:
  - `_classify_broad_sector(code)` — 宽基行业分类 (从 strategy_reversal.py 复制)
  - `load_industry_map(engine)` — 从 MySQL daily_signal_detail 加载 code→industry
  - `compute_sector_heat_c(flat_df, industry_map)` — 计算 sector_boost，返回 DataFrame
- **新增配置**:
  ```python
  SECTOR_HEAT_ENABLED = True
  SECTOR_HEAT_HARD_FILTER = 0.65
  SECTOR_HEAT_MIN = 0.5
  SECTOR_HEAT_MAX = 1.5
  ```
- **数据流**:
  1. 加载 kline → 构建 code_klines {code: DataFrame}
  2. 构建 flat_df: code, trade_date, close (从 code_klines 提取)
  3. 计算 ret_1d per code
  4. `load_industry_map()` → {code: industry}, 不足时 `_classify_broad_sector` 回退
  5. `compute_sector_heat_c()` → DataFrame [trade_date, code, sector_boost]
  6. 构建 `sector_heat_cache = {date: {code: sector_boost}}`，日期偏移1天(无未来)
  7. 候选选择循环中：`sector_heat_cache.get(next_date, {}).get(code, 1.0) < 0.65 → skip`
- **理由**: 避开过热板块，降低追高风险，提升胜率
- **替代方案**: 只用宽基行业分类（无MySQL），但精度低；选择保留 MySQL + 宽基回退
- **风险**: MySQL 不可用 → SECTOR_HEAT_ENABLED=False 降级；宽基 fallback 覆盖全量
