## Design Decisions

### Decision 1: Leverage implementation
**Choice**: Use `leverage_multiplier` as NAV-only multiplier (not position sizing)
**Rationale**: BacktestEngine already uses `leverage` for NAV calc. Adding a second multiplier on top is clean and doesn't require tracking per-position capital. The base `leverage` stays 1.0 (iron law).
**Alternative**: Position sizing via capital_per_stock = capital/top_n * multiplier — but this adds complexity without benefit in the current engine.
**Risk**: NAV multiplier achieves same mathematical effect as position sizing for backtest purposes.

### Decision 2: Daily rebalance implementation
**Choice**: Add `rebalance_freq` config ("monthly"|"daily"), modify `is_rebalance_day` logic
**Rationale**: Avoids messing with the weekly rebalance flag; clean extension.
**Implementation**: When `rebalance_freq == "daily"`, every trading day triggers rebalance. When "monthly", existing monthly logic.
**Daily selection**: re-rank every day, keep holdings that stay in top N, only replace those that drop out.
**Alternative**: Weekly rebalance (3rd option) — not needed per task spec.
**Risk**: Higher turnover costs; task acknowledges this and accepts it.

### Decision 3: Stop loss adjustment
**Choice**: `individual_stop: -0.10 → -0.12` for 1.5x leverage
**Rationale**: Task body specifies this. -12% with 1.5x = -18% effective, wider than -8% at 1x but provides breathing room.
**Risk**: Larger individual losses; compensated by daily rebalance removing weak holdings faster.

### Decision 4: Max daily drawdown
**Choice**: `max_daily_drawdown: -0.15`, force halve positions at end of day
**Rationale**: Task body: "单日-15%强制减半". This is a portfolio-level circuit breaker.
**Implementation**: After NAV calculation each day, if `day_ret < max_daily_drawdown`, halve all positions at next day's open.

### Decision 5: PolicyWeight daily update
**Choice**: When rebalance_freq == "daily", PolicyWeight updates daily (half-life still 5 days)
**Rationale**: Task body: "日频调仓时PolicyWeight更新频率也改为日频"
